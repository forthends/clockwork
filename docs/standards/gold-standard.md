# 黄金标准（Gold Standard）

> 本文件定义了 Clockwork 治理框架下所有产物的质量基线。所有 Agent 在产出和校验产物时，必须以此为标准。

---

## 通用质量标准

以下标准适用于所有类型的产物文档：

### 结构完整性

- 必须基于对应模板创建，保留所有必填章节
- 每个章节必须有实质内容，不得仅有标题或占位符
- 变更记录章节必须随文档更新而更新

### 内容准确性

- 引用上游产物的内容必须准确，不得编造
- 数据和指标必须可追溯到来源
- 术语使用必须一致，遵循项目词汇表

### 可理解性

- 使用清晰、简洁的语言
- 避免歧义表达，关键概念需给出明确定义
- 复杂逻辑需配以图表辅助说明

### 可追溯性

- 下游产物必须可追溯到上游产物
- 测试用例追溯到验收标准，验收标准追溯到用户故事
- 设计决策必须记录选择理由

## PRD 质量标准

| 维度 | 标准 | 判定方式 |
|------|------|---------|
| 背景描述 | 清晰阐述业务场景和需求动机 | 新读者能在不询问的情况下理解为什么要做这个功能 |
| 目标定义 | 目标可量化，有明确的成功指标 | 包含具体的数字或可验证的条件 |
| 范围界定 | 明确包含和不包含的范围 | 不存在 "以后再说" 的模糊地带 |
| 用户故事 | 覆盖所有功能点，格式规范 | 每个独立用户行为路径都有对应故事 |
| 验收标准 | 每个故事至少一条，可量化可验证 | QA 能直接基于验收标准编写测试用例 |

---

## 技术设计质量标准

| 维度 | 标准 | 判定方式 |
|------|------|---------|
| 架构方案 | 方案完整，技术选型有理由 | 包含架构图，每个选型说明了 "为什么" |
| 设计决策 | 记录备选方案和选择理由 | 不存在 "就这样吧" 的无理由决策 |
| API 设计 | 覆盖所有功能点，格式完整 | 每个接口有请求/响应/错误码定义 |
| 数据模型 | 字段类型和约束明确 | 开发者能直接基于文档建表 |
| 风险识别 | 识别技术风险和缓解措施 | 不存在 "到时候再看" 的未处理风险 |

---

## 测试计划质量标准

| 维度 | 标准 | 判定方式 |
|------|------|---------|
| 测试范围 | 明确范围内外和测试策略 | 不存在 "可能需要测" 的模糊表述 |
| 用例覆盖率 | 每条验收标准至少一个用例 | AC → TC 映射表无遗漏 |
| 用例质量 | 包含前置条件、步骤和预期结果 | 任何人都能按步骤执行并判定结果 |
| 异常覆盖 | 覆盖错误输入和边界条件 | 不仅测 "能跑通" 还测 "跑不通时怎么办" |
| 环境定义 | 环境配置和测试数据方案明确 | 能独立搭建测试环境并准备数据 |

---

## 代码评审质量标准

| 维度 | 标准 | 判定方式 |
|------|------|---------|
| 功能完整性 | 代码覆盖技术设计中所有功能点 | 逐项对照检查清单 |
| 代码质量 | 命名清晰、职责单一、无重复 | 代码自解释，读者无需猜测意图 |
| 错误处理 | 边界条件和异常路径有处理 | 不存在 "这里不会出错" 的假设 |
| 安全性 | 无明显安全漏洞 | 输入校验、权限控制到位 |
| 意见分级 | 评审意见标注等级和修复方向 | 开发者知道哪些必须改、哪些建议改 |

---

## 校验未通过的处理

当产物未通过质量校验时：

1. **定位问题**：根据校验报告确定未通过的具体项
2. **修正内容**：针对性修改产物，而非推倒重来
3. **重新校验**：修改后再次调用 `validate-artifact` 技能
4. **记录迭代**：在变更记录中注明修正内容

反复校验未通过（>3 次）时，应回溯上游产物是否存在根本性问题。
