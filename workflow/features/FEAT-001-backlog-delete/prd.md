> **需求名称**：Backlog 列表删除功能  
> **创建日期**：2026-04-10

> **状态**：已定稿

## overview | 需求概述

### 背景

AgileHub 的需求池管理（Backlog）模块目前支持需求的创建、查看和编辑操作，后端已提供 `DELETE /backlog/{backlog}` 接口，但前端 Backlog 列表页面（`backlog/index.tsx`）尚未暴露删除入口。用户在录入需求后，如果发现需求重复或录入错误，无法直接在列表页中删除，只能通过其他方式处理，影响日常使用效率。

### 目标

- 在 Backlog 列表页面为用户提供单条需求的删除能力
- 通过状态约束和确认对话框保障数据安全，防止误删已进入交付流程的需求
- 复用已有的后端 DELETE API，保持前后端一致性

### 范围

- **包含**：
  - 在 Backlog 列表每行的操作菜单中增加"删除"选项
  - 点击删除后弹出确认对话框
  - 仅允许删除状态为 `pending` 的需求
  - 非 `pending` 状态的需求，删除选项置灰或隐藏
  - 删除成功后刷新列表，显示成功提示
  - 删除失败时显示错误提示
- **不包含**：
  - 批量删除功能
  - 需求回收站 / 软删除恢复
  - 需求详情页的删除入口

## user_stories | 用户故事

### 故事 1：删除误录入的待处理需求

- **作为** 团队成员
- **我想要** 在 Backlog 列表中删除一条状态为 `pending` 的需求
- **以便** 清理重复或错误录入的需求条目，保持需求池整洁

**补充说明**：用户通过列表行末尾的操作菜单（"⋯" 按钮）进入删除操作，点击后需弹出确认对话框，确认后才执行删除。

### 故事 2：防止误删已进入流程的需求

- **作为** 团队成员
- **我想要** 系统阻止我删除已经进入交付流程（非 `pending` 状态）的需求
- **以便** 避免破坏正在进行中的 Sprint 任务链和关联数据

**补充说明**：对于状态不是 `pending` 的需求，操作菜单中的"删除"选项应不可用（置灰并附带提示，或直接隐藏），让用户明确理解约束原因。

## acceptance_criteria | 验收标准

### AC-1：Pending 状态需求可成功删除

- **前置条件**：Backlog 列表中存在至少一条状态为 `pending` 的需求
- **操作**：用户点击该需求行末尾的操作菜单（"⋯"），选择"删除"选项
- **预期结果**：弹出确认对话框，内容包含需求标题，提示"确认要删除这条需求吗？删除后不可恢复。"

### AC-2：确认删除后需求从列表移除

- **前置条件**：确认对话框已弹出
- **操作**：用户点击"确认删除"按钮
- **预期结果**：
  - 调用 `DELETE /backlog/{backlog}` 接口
  - 删除成功后，列表自动刷新，该需求不再显示
  - 页面显示成功提示（如 toast 通知："需求已删除"）

### AC-3：取消删除保持列表不变

- **前置条件**：确认对话框已弹出
- **操作**：用户点击"取消"按钮或关闭对话框
- **预期结果**：对话框关闭，列表数据无任何变化

### AC-4：非 Pending 状态需求不可删除

- **前置条件**：Backlog 列表中存在状态为 `parsing`、`sprint_planned`、`in_progress` 或 `done` 的需求
- **操作**：用户查看该需求行的操作菜单
- **预期结果**：删除选项不可操作（置灰显示并附带提示文案"仅待处理状态的需求可删除"，或直接隐藏删除选项）

### AC-5：删除失败时给予错误反馈

- **前置条件**：用户确认删除一条 `pending` 状态的需求
- **操作**：后端接口返回错误（如网络异常、权限不足、需求已被其他操作关联）
- **预期结果**：
  - 对话框或页面显示错误提示（如 toast 通知："删除失败，请稍后重试"）
  - 列表数据保持不变，不会出现数据丢失假象

## 补充信息

### 非功能性需求

- **性能**：删除操作响应时间应在 1 秒内完成（正常网络条件下）
- **安全**：后端需校验需求状态，即使前端绕过限制直接调用 API 删除非 `pending` 需求，后端也应拒绝并返回 422 错误
- **兼容性**：与现有 Backlog 列表的筛选、分页功能兼容，删除后分页状态应合理更新

### 依赖与风险

| 依赖/风险 | 描述 | 影响 | 缓解措施 |
|-----------|------|------|---------|
| 后端 API 已存在 | `DELETE /backlog/{backlog}` 接口已实现 | 正面：减少后端开发量 | 需确认后端是否已有状态校验逻辑，若无则需补充 |
| 关联数据完整性 | 需求可能有附件（attachments） | 删除需求时附件文件是否一并清理 | 需开发团队确认后端删除逻辑是否处理附件级联 |
| 前端页面缺口 | 项目简介提到 `backlog/show.tsx` 缺失 | 本需求不涉及详情页，无直接影响 | 后续可单独处理 |

### 优先级

- **紧急程度**：中
- **业务价值**：中
- **实现复杂度**：低
