# 校验产物质量

---

## 技能信息

- **技能名称**：校验产物质量
- **技能标识**：`validate-artifact`
- **一句话说明**：对产物进行结构完整性和内容质量双重校验，确保符合质量闸门标准
- **适用角色**：PM / Developer / Tester / Reviewer（所有角色通用）

---

## 触发条件

当以下条件满足时，应调用本技能：

- 某个阶段的产物编写完成，需要进行质量校验
- 产物修改后需要重新校验
- 流程推进前的最后检查

---

## 前置条件

调用本技能前，必须确保：

1. 产物文件已创建并保存到 `workflow/features/FEAT-<id>-<name>/` 目录
2. 工作流定义文件存在于 `workflow/_definitions/` 目录
3. manifest.yaml 中该阶段 `status` 为 `in_progress`

---

## 校验类型

本技能执行三类校验，层层递进：

| 校验类型 | 说明 | 失败后果 |
|----------|------|----------|
| **S1 - 结构校验** | 产物是否包含所有必需章节 | 阻塞 → 必须修复 |
| **S2 - 内容校验** | 章节内容是否有实质意义（非空占位符） | 阻塞 → 必须修复 |
| **S3 - 引用校验** | 跨文件引用是否准确存在 | 阻塞 → 必须修复 |

---

## 执行步骤

### 第一步：读取上下文

1. 读取 `workflow/features/FEAT-<id>-<name>/manifest.yaml`
2. 从 manifest 获取 `workflow` 名称和当前阶段 `id`
3. 读取 `workflow/_definitions/<workflow>.yaml`
4. 定位当前阶段的 `outputs[].validation` 规则

### 第二步：S1 - 结构校验

读取产物文件，按 `required_sections` 逐项检查：

```
检查清单：
[ ] 章节是否存在（按 `##` 或 `###` 标题识别）
[ ] 章节是否为空（仅有标题无正文内容）
```

**失败条件**：任何 required_section 缺失或为空

### 第三步：S2 - 内容校验

对每个已通过的章节，进行深度内容检查：

#### 通用规则（所有产物适用）

| 规则 | 检查逻辑 | 失败原因示例 |
|------|----------|--------------|
| `no_placeholder` | 内容不能仅含 `<占位符>` 模式 | 发现 "`<描述需求产生的背景>`" 等未填充占位符 |
| `no_lorem` | 不能包含 lorem ipsum 等占位文本 | 发现 "Lorem ipsum dolor sit amet" |
| `meaningful_length` | 正文内容至少 50 字符 | 章节存在但内容仅 20 字符 |

#### 领域规则（按产物类型启用）

**PRD (prd.md) 专用规则**：

| 规则 | 检查逻辑 | 失败原因示例 |
|------|----------|--------------|
| `ac_actionable` | acceptance_criteria 每条必须包含"操作"和"预期结果" | AC 条目缺少"操作"或"预期结果"字段 |
| `ac_verifiable` | 验收标准必须可验证（不能是主观描述） | "界面美观" 等无法客观验证的描述 |
| `story_complete` | 每个 user_story 必须包含"作为/我想要/以便"三要素 | 故事缺少"以便"字段 |
| `scope_boundary` | 必须明确区分"包含"和"不包含" | 范围界定模糊或缺失"不包含"部分 |

**技术设计 (tech-design.md) 专用规则**：

| 规则 | 检查逻辑 | 失败原因示例 |
|------|----------|--------------|
| `has_api_contract` | api_design 章节必须定义端点、请求、响应 | API 设计缺少请求/响应格式定义 |
| `has_data_model` | data_model 章节必须包含实体关系或表结构 | 数据模型缺失或过于简略 |
| `no_impl_details` | architecture 章节不能包含具体代码实现 | 发现具体的函数实现代码 |

**测试计划 (test-plan.md) 专用规则**：

| 规则 | 检查逻辑 | 失败原因示例 |
|------|----------|--------------|
| `has_test_cases` | test_cases 章节必须有具体测试用例 | 仅有测试目标无具体用例 |
| `case_traceable` | 每个用例必须可追溯到对应的 AC | 用例未关联验收标准 |

### 第四步：S3 - 引用校验

对于声明了 `inputs` 的阶段，校验上游产物的引用准确性：

1. 从 workflow YAML 读取当前阶段的 `inputs[]`
2. 对每个 input，解析 `from: "<stage>.<output>"` 格式
3. 校验引用的文件是否存在于 `workflow/features/FEAT-<id>-<name>/` 目录
4. 校验引用内容是否在产物中被正确提及（如引用了某上游结论）

**跨文件引用校验示例**：

```
当前阶段：tech_design
inputs:
  - from: requirements.prd

校验：
[ ] requirements/prd.md 文件是否存在
[ ] tech-design.md 中是否引用了 PRD 中的关键结论
[ ] 引用的章节/表格/数据是否在 PRD 中真实存在
```

### 第五步：执行业务规则（如有）

读取 `validation.custom_rules`，逐条执行：

```yaml
custom_rules:
  - "代码必须在 repos/ 中对应的代码仓库内提交"
  - "提交信息必须关联需求编号"
```

每条规则需要：
1. 解析规则描述为可执行的检查逻辑
2. 读取相关上下文（如 git log、文件内容）
3. 判定是否满足

### 第七步：更新 manifest

根据校验结果更新 manifest.yaml：

**校验通过**：
```yaml
stages:
  requirements:
    status: completed
    completed_at: 2026-04-18T14:30:00
    validation_passed: true
    artifacts:
      - prd.md
```

**校验未通过**：
```yaml
stages:
  requirements:
    status: in_progress
    validation_passed: false
    notes: |
      2026-04-18 校验未通过:
      - AC-3 缺少"操作"字段
      - AC-5 "界面美观" 无法客观验证
```

---

## 输入

| 输入 | 来源 | 说明 |
|------|------|------|
| 待校验产物 | `workflow/features/FEAT-<id>-<name>/<artifact>.md` | 需要校验的产物文件 |
| 工作流定义 | `workflow/_definitions/<workflow>.yaml` | 包含 validation 规则 |
| 实例状态 | `workflow/features/FEAT-<id>-<name>/manifest.yaml` | 当前流程状态 |

---

## 输出

| 输出 | 形式 | 说明 |
|------|------|------|
| 校验报告 | 控制台输出 | 展示校验结果和未通过项及修正指引 |
| manifest 更新 | 文件更新 | 更新阶段状态和校验结果 |

---

## 注意事项

- **校验必须严格**：不得放宽标准放行不合格产物
- **修正指引必须具体**：指出具体问题所在和如何修复
- **同一产物可多次校验**：直到通过为止
- **校验通过是流程推进的必要条件**：未通过不得进入下一阶段
- **S3 引用校验仅在有 inputs 时执行**：无上游输入时跳过
